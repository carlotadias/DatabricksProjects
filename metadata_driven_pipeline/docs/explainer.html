End-to-End Architecture

Everything runs natively in Databricks. A YAML config defines all sources. Auto Loader reads files from ADLS and writes to Delta tables. One Lakeflow Job orchestrates the full pipeline.

Auto Loader = ADLS to Delta Tables Auto Loader reads files from ADLS paths and writes to Delta tables. It tracks which files have been processed via checkpoints — no custom file tracking needed. Used twice: once to encrypt staging files into raw (binary read), and once inside the Lakeflow pipeline to decrypt and parse into tables (json read).
Encryption keys In production, encryption keys live in Azure Key Vault, accessed via dbutils.secrets.get(scope, key). The Databricks Secrets scope is backed by Key Vault — keys never appear in code or config files.

Onboard a New Pipeline

All generic notebooks are reused as-is. You only write config and (optionally) a domain-specific joins notebook.

1

Create sources.yml for the new domain

Define ADLS paths, encryption settings, and one block per source with schema and PII columns.

# conf/sources.yml
settings:
  staging_root: abfss://data@mydatalake.dfs.core.windows.net/staging
  raw_root: abfss://data@mydatalake.dfs.core.windows.net/raw
  ad_group: my-pii-authorized-group
  encryption_key: "USE_SECRETS_IN_PROD"  # prod: dbutils.secrets.get(scope, key)

sources:
  - name: meter_readings
    prefix: smart_meters/readings
    format: json
    schema: "meter_id STRING, timestamp TIMESTAMP, reading DOUBLE, unit STRING"
    bronze_columns: [meter_id, timestamp, reading, unit]
    pii_columns: [customer_address]
    join_key: meter_id
    pdv_table: pdv_meter_readings
2

Create databricks.yml

This is the full file. Only 3 variables (environment-level), then two resources: a Lakeflow pipeline and a job.

# ──────────────────────────────────────────────────────────
# VARIABLES — only what changes between dev / prod
# ──────────────────────────────────────────────────────────
bundle:
  name: smart_meter_pipeline

variables:
  catalog:                          # REQUIRED — Unity Catalog catalog
    default: my_catalog
  config_schema:                    # REQUIRED — schema for the config table
    default: smart_meter_demo
  bronze_schema:                    # REQUIRED — target schema for bronze tables
    default: smart_meter_bronze

# ──────────────────────────────────────────────────────────
# LAKEFLOW PIPELINE — pick which reusable notebooks to include
# ──────────────────────────────────────────────────────────
resources:
  pipelines:
    smart_meter_lakeflow:
      name: smart_meter_lakeflow
      serverless: true
      catalog: ${var.catalog}
      target: ${var.bronze_schema}
      libraries:                        # ALWAYS REQUIRED:
        - notebook:
            path: ./notebooks/pipeline/ingest_and_decrypt.py   # decrypts + parses
        - notebook:
            path: ./notebooks/pipeline/bronze_tables.py        # creates bronze tables
                                          # OPTIONAL — add if needed:
                                          #   pdv_tables.py         → if PII columns exist
                                          #   derived_joins.py      → if cross-source joins needed
      configuration:
        config_table: "${var.catalog}.${var.config_schema}.pipeline_config"

# ──────────────────────────────────────────────────────────
# JOB — 3 tasks always present, 4th (apply_masks) optional
# ──────────────────────────────────────────────────────────
  jobs:
    smart_meter_pipeline:
      name: smart_meter_pipeline
      tasks:
        - task_key: load_config           # ALWAYS — reads sources.yml → Delta config table
          notebook_task:
            notebook_path: ./notebooks/setup_config.py
            base_parameters:
              catalog: "${var.catalog}"
              config_schema: "${var.config_schema}"
              bronze_schema: "${var.bronze_schema}"
              config_file: "sources.yml"     # ← links to conf/sources.yml
              force_refresh: "false"

        - task_key: encrypt_staging_to_raw  # ALWAYS — Auto Loader: ADLS staging → ADLS raw
          depends_on: [load_config]
          notebook_task:
            notebook_path: ./notebooks/encrypt_staging_to_raw.py
            base_parameters:
              config_table: "${var.catalog}.${var.config_schema}.pipeline_config"

        - task_key: run_pipeline           # ALWAYS — Lakeflow: ADLS raw → Bronze Delta tables
          depends_on: [encrypt_staging_to_raw]
          pipeline_task:
            pipeline_id: ${resources.pipelines.smart_meter_lakeflow.id}

                                          # OPTIONAL — add if PII columns exist:
                                          #   apply_column_masks task (runs apply_column_masks.py)

# ──────────────────────────────────────────────────────────
# TARGETS — set variable values per environment
# ──────────────────────────────────────────────────────────
targets:
  dev:
    default: true
    workspace:
      host: https://my-dev-workspace.cloud.databricks.com
    variables:
      catalog: dev_catalog
      config_schema: smart_meter_demo
      bronze_schema: smart_meter_bronze
  prod:
    workspace:
      host: https://my-prod-workspace.cloud.databricks.com
    variables:
      catalog: prod_catalog
      config_schema: smart_meter
      bronze_schema: smart_meter_bronze
3

Deploy and run

databricks bundle deploy
databricks bundle run smart_meter_job
Adding a source to an existing pipeline Even simpler: add a block to the existing sources.yml, redeploy, and run the pipeline with force_refresh=true to reload config. No notebook changes. No table schema changes.

Configuration

All pipeline config lives in a single YAML file, version-controlled in git. A setup notebook loads it into a Delta table with a flexible MAP schema.

sources.yml structure

settings:                               # Pipeline-level config
  staging_root: abfss://data@mydatalake.dfs.core.windows.net/staging
  raw_root: abfss://data@mydatalake.dfs.core.windows.net/raw
  ad_group: my-pii-authorized-group
  encryption_key: "..."               # prod: dbutils.secrets.get(scope, key)

sources:                                # One block per data source
  - name: contact_records
    prefix: domain/source_name   # subfolder within the container
    format: json
    schema: "contact_id STRING, channel STRING, ..."
    bronze_columns: [contact_id, channel, outcome, ...]
    pii_columns: [customer_phone, customer_name]
    join_key: contact_id
    pdv_table: pdv_contact_records

Delta config table (flexible schema)

CREATE TABLE pipeline_config (
  source_name  STRING                -- unique key ("_settings" for pipeline-level)
  is_active    BOOLEAN               -- enable/disable without deleting
  properties   MAP<STRING, STRING>   -- all config as key-value pairs
)
Why MAP instead of fixed columns? If a future pipeline needs partition_by, watermark_column, or merge_keys — just add the property to the YAML. The MAP accepts it without schema changes.
Same ADLS container, different prefixes All sources land in the same ADLS container but each declares its own prefix (subfolder) — e.g. domain/source_name. The setup notebook assembles full paths: staging_root/prefix and raw_root/prefix.

Reusable Notebooks

Every notebook reads the pipeline_config Delta table and dynamically handles whichever sources are active. No notebook knows about specific sources — everything is config-driven.

encrypt_staging_to_raw.py
Auto Loader reads new files from ADLS staging → encrypts content with AES-GCM → writes encrypted files to ADLS raw. One stream per source, fault-tolerant.
# Encryption key from config (prod: from Azure Key Vault via Secrets)
encryption_key = dict(settings_row.properties)["encryption_key"]

for source in active_sources:
    staging_path = props["staging_path"]   # e.g. abfss://data@.../staging/amazon_connect/contact_records
    raw_path     = props["raw_path"]       # e.g. abfss://data@.../raw/amazon_connect/contact_records

    spark.readStream.format("cloudFiles")
        .option("cloudFiles.format", "binaryFile")  # reads entire file as bytes
        .load(staging_path)                           # Auto Loader watches ADLS path
        .withColumn("encrypted_content",
            base64(aes_encrypt(content, key, 'GCM')))
        .writeStream
        .option("checkpointLocation", f"{raw_path}/_checkpoint")
        .option("path", raw_path)               # writes encrypted files to ADLS raw
        .trigger(availableNow=True).start()
pipeline/ingest_and_decrypt.py
Inside Lakeflow pipeline — Auto Loader reads encrypted files from ADLS raw → decrypts in-memory → parses JSON → creates temp raw_* tables.
for cfg in active_sources:
    raw_path = props["raw_path"]           # ADLS path to encrypted files
    schema   = StructType.fromDDL(props["schema"])

    @dlt.table(name=f"raw_{source_name}", temporary=True)
    def create_raw():
        return (spark.readStream.format("cloudFiles")
            .option("cloudFiles.format", "json")
            .option("rescuedDataColumn", "_rescue")
            .load(raw_path)                       # Auto Loader watches ADLS raw path
            .withColumn("decrypted_json",
                cast(aes_decrypt(unbase64(encrypted_content), key, 'GCM') as STRING))
            .withColumn("json_line", explode(split(col("decrypted_json"), "\n")))
            .withColumn("parsed", from_json(col("json_line"), schema))
            .select("parsed.*", "source_file", "source_modified"))
pipeline/bronze_tables.py
Reads bronze_columns from config → creates bronze_* Delta streaming tables with data quality expectations.
for cfg in active_sources:
    bronze_cols = json.loads(props["bronze_columns"])

    @dlt.table(name=props["bronze_table"])
    @dlt.expect_or_drop("valid_record", f"{bronze_cols[0]} IS NOT NULL")
    def create_bronze():
        return dlt.read_stream(f"raw_{src}").select(*bronze_cols, "source_file", "_ingested_at")
pipeline/pdv_tables.py
Reads pii_columns from config → encrypts each PII column with AES-GCM → persists as PDV Delta tables.
for cfg in active_sources:
    pii_cols  = json.loads(props["pii_columns"])
    pdv_table = props["pdv_table"]

    @dlt.table(name=pdv_table)
    def create_pdv():
        df = dlt.read_stream(f"raw_{src}")
        for c in pii_cols:
            df = df.withColumn(c, base64(aes_encrypt(col(c), key, 'GCM')))
        return df.select(join_key, *pii_cols)
apply_column_masks.py
Attaches UC Column Masks to PDV tables → creates serving views that JOIN bronze + PDV. AD group members see decrypted PII, everyone else sees REDACTED.
# Column mask: decrypts for authorised users, redacts for everyone else
ALTER TABLE {pdv_table} ALTER COLUMN {col}
    SET MASK pii_decrypt

# Serving view: joins bronze (no PII) with PDV (encrypted PII)
CREATE VIEW serving_{source} AS
    SELECT b.*, p.customer_phone, p.customer_name
    FROM bronze_{source} b JOIN pdv_{source} p USING ({join_key})
The pattern is always the same Every notebook: (1) reads config table, (2) loops over active sources, (3) uses properties from the MAP column. Zero hardcoded source names, schemas, or paths. Reuse across any domain pipeline.

Fault Tolerance

If a file arrives malformed, the pipeline does not crash. Bad data is filtered, good data is processed, and everything is visible in native Databricks logs.

What happens to bad data

LayerMechanismWhat happens
Encryption Per-source try/except If one source fails, others continue. Failures are logged.
Ingest from_json + split Each JSON line is parsed individually. Invalid JSON or wrong-schema records produce NULL fields.
Bronze expect_or_drop Rows with NULL primary keys (complete parse failures) are dropped. Pass/fail counts logged in event log.
Audit Pipeline event log Every table's expectation results (passed, dropped) are automatically logged per run.

How to inspect after a failure

Run inspect_pipeline_health.py — it queries the pipeline event log:

-- See which tables had dropped rows and how many
SELECT
    timestamp,
    origin.flow_name AS table_name,
    details:flow_progress.metrics.num_output_rows AS output_rows,
    details:flow_progress.data_quality.dropped_records AS dropped_rows
FROM event_log(TABLE(catalog.schema.bronze_my_source))
WHERE event_type = 'flow_progress'
  AND details:flow_progress.data_quality IS NOT NULL

-- See which files were processed per table
SELECT DISTINCT source_file, source_modified, count(*) AS rows
FROM bronze_my_source
GROUP BY source_file, source_modified

Restart after failure

Auto Loader checkpoints make restart automatic
  1. Fix the file in ADLS (correct the format, or remove the bad file)
  2. Re-run the pipeline — Auto Loader picks up only unprocessed files
  3. Successfully processed files from the same batch are not re-ingested
No manual checkpoint manipulation. No MetaDB updates. No code changes.

Advanced: resetting the checkpoint to reprocess files

In rare cases (e.g. a bad encryption key was used and all raw files need re-encryption), you may need to tell Auto Loader to reprocess files from scratch or from a specific point.

ScenarioActionCommand / Code
Reprocess all files from scratch Delete the checkpoint directory 
dbutils.fs.rm("abfss://data@.../raw/source/_checkpoint", True)
# Then re-run the pipeline — Auto Loader will re-discover all files
Reprocess files modified after a specific time Use modifiedAfter option 
spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("modifiedAfter", "2025-06-01T00:00:00.000Z")
    .load(raw_path)
Lakeflow pipeline full refresh Run pipeline with full refresh flag 
# From CLI:
databricks bundle run my_pipeline --params run_pipeline.full_refresh=true

# Or in databricks.yml (temporary):
pipeline_task:
  pipeline_id: ${resources.pipelines.connect_pipeline.id}
  full_refresh: true
When NOT to delete checkpoints Deleting a checkpoint means Auto Loader loses track of which files were already processed. If the downstream tables still contain the old data, you may end up with duplicate rows. Always pair a checkpoint reset with either:
  • A full refresh of the Lakeflow pipeline (recreates all tables), or
  • A MERGE strategy in the downstream tables (deduplicates on primary key)
Checkpoint locations
  • encrypt_staging_to_raw: checkpoint is at {raw_path}/_checkpoint — you control this path directly
  • Lakeflow pipeline (ingest_and_decrypt): checkpoints are managed internally by Lakeflow — use full refresh to reset them

Schema Evolution

What happens when your source adds a column, removes one, or renames a field?

Summary — three approaches compared

Scenario Declarative Pipeline
+ explicit schema from config
(approach used here)		 Declarative Pipeline
+ Auto Loader schema inference		 Plain notebook
(spark.readStream → writeStream)
Add a column Incremental — update sources.yml, redeploy, no full refresh needed Full refresh required — new data goes to _rescued_data Automatic — addNewColumns + mergeSchema
Remove a column No action — column becomes NULL No action — column becomes NULL No action — column becomes NULL
Rename a column Full refresh + upstream must change field name Full refresh + upstream must change field name Manual code change (COALESCE / alias)
SQL ALTER TABLE RENAME COLUMN Not supported on streaming tables Supported (you own the table)
Why Declarative Pipeline + explicit schema handles add-column better Standard Declarative Pipelines use Auto Loader schema inference — the table schema is fixed at creation and new columns land in _rescued_data. Our pipeline uses an explicit schema from config (from_json(json_line, schema)). When you update the schema in sources.yml and redeploy, the from_json parse includes the new column in the DataFrame, and Lakeflow evolves the streaming table schema automatically on the next incremental run.

Adding a column (incremental, no full refresh)

When upstream starts sending a new field:

  1. Add the new column to sources.yml — both in schema and bronze_columns
  2. Redeploy with force_refresh=true (reloads the config table)
  3. Run the pipeline incrementally (no full refresh needed)

The new column appears in the table immediately. Old rows have NULL, new rows have the value. No data loss.

How it works

# ingest_and_decrypt.py — schema comes from config, not file inference
schema = StructType.fromDDL(props["schema"])  # updated schema includes the new column

@dlt.table(name=f"raw_{source_name}", temporary=True)
def create_raw():
    return (spark.readStream.format("cloudFiles")
        .load(raw_path)
        .withColumn("parsed", from_json(col("json_line"), schema))  # ← new column parsed here
        .select("parsed.*", ...))

Because we control the schema via config, Lakeflow sees the new column in the returned DataFrame and evolves the table automatically.

Renaming a column

SQL ALTER TABLE — not supported

ALTER TABLE catalog.schema.bronze_my_source
    RENAME COLUMN old_name TO new_name
Not supported on streaming tables [STREAMING_TABLE_OPERATION_NOT_ALLOWED.UNSUPPORTED_OPERATION]
Declarative Pipeline streaming tables are managed by the pipeline framework. Direct DDL operations like RENAME COLUMN are blocked.

Config-only change — does not work either

If you update sources.yml with the new column name but the source files still use the old name:

  • from_json looks for a JSON key matching the new name
  • The source files don't have that key — the column is all NULLs
  • The old column name disappears from the schema

Correct approach

  1. Upstream changes the field name in the source data
  2. Update sources.yml with the new name in schema, bronze_columns, and any derived_joins.py references
  3. Run a full refresh of the pipeline (recreates all tables from scratch)
A rename is a coordinated change Column renames require both the data producer (upstream source) and the pipeline config to change simultaneously. This is expected for production systems — renames are breaking changes that should be planned.

Declarative Pipeline + schema inference — for reference

If you use a Declarative Pipeline without an explicit schema (relying on Auto Loader schema inference), the behaviour is different:

Adding a column

  • Auto Loader puts the new field into _rescued_data (a JSON string). The pipeline doesn't break but the column is not added to the table.
  • You can extract values with get_json_object(_rescued_data, '$.new_field') as a workaround.
  • To get a proper typed column: update the schema definition + full refresh.

Removing a column

  • The column stays in the table with NULLs for new rows. No action needed.

Plain notebooks — Auto Loader schema evolution

If you use Auto Loader in a plain notebook (not Lakeflow), you manage everything yourself. This gives you the most control over schema evolution:

# Plain notebook — Auto Loader with schema evolution enabled
(spark.readStream
    .format("cloudFiles")
    .option("cloudFiles.format", "csv")
    .option("cloudFiles.schemaLocation", checkpoint_path)       # you manage schema storage
    .option("cloudFiles.schemaEvolutionMode", "addNewColumns") # auto-detect new columns
    .load(source_path)
    .writeStream
    .option("checkpointLocation", checkpoint_path)              # you manage checkpoints
    .option("mergeSchema", "true")                              # you tell Delta to accept changes
    .table("bronze_customers")
)
OptionLakeflow manages itPlain notebook

schemaEvolutionModeAutomaticYou set it
checkpointLocationAutomaticYou manage it
mergeSchemaAutomaticYou set "true"

In a plain notebook, addNewColumns detects new columns in files, restarts the stream, and adds them to the Delta table automatically. You also get ALTER TABLE RENAME COLUMN because you own the table directly.

Detecting schema drift

Monitor the rescue column to know when new columns start arriving before your config is updated:

-- Which tables have unexpected columns arriving?
SELECT
    'bronze_my_source' AS table_name,
    COUNT(*) AS rows_with_rescued_data,
    collect_set(json_object_keys(_rescued_data)) AS new_columns
FROM bronze_my_source
WHERE _rescued_data IS NOT NULL
    AND _rescued_data != '{}'

If this returns results, new columns are arriving that aren't in your sources.yml schema yet. Add them to the config and redeploy — no full refresh needed with our approach.
